## Aturan yang Diterapkan

1. Admin/penjual bisa atur **rasio poin** (contoh: Rp1.000 = 1 poin) lewat form, bukan hardcode.
2. Ada **minimal nominal transaksi** supaya dapat poin (bisa diset ke 0 kalau tidak ada minimal).
3. Aturan poin punya status **aktif/nonaktif** — perhitungan poin transaksi memakai aturan yang sedang `aktif`.
4. Setiap perubahan saldo poin (dapat/pakai) dicatat di tabel `riwayat_poin`, dengan jenis `masuk` atau `keluar`, supaya bisa diaudit/dilacak.
5. Perhitungan poin (`hitungPoin()`) murni fungsi backend — nominal transaksi masuk, poin keluar dihitung server-side (selaras dengan aturan project: perhitungan ulang wajib di backend, tidak boleh percaya angka dari frontend).

## Method Integrasi Transaksi (siap dipanggil modul Transaksi)

Semua ada di `RiwayatPoinModel.php`:

| Method | Kapan dipanggil | Keterangan |
|--------|------------------|------------|
| `getSaldoPoin(int $userId): int` | Kapan saja | Ambil saldo poin user saat ini. |
| `prosesTambahPoin(int $userId, int $jumlahPoin, ?int $transaksiId, string $keterangan)` | Setelah transaksi sukses (status `5 = Active` / `20 = Terposting`) | Tambah poin + catat riwayat + update saldo di tabel `user`. Hitung `$jumlahPoin` pakai `PoinModel::hitungPoin($totalBelanja)` dulu. |
| `prosesPakaiPoin(int $userId, int $jumlahPoin, ?int $transaksiId, string $keterangan)` | Saat checkout, kalau pembeli pilih pakai saldo poin (Multipayment) | Return `false` kalau saldo tidak cukup (tidak memotong apapun). Cek nilai baliknya sebelum lanjut proses pembayaran. |
| `rollbackPoinByTransaksi(int $transaksiId)` | Saat transaksi dibatalkan (status `8 = Cancel`) | Otomatis kembalikan/batalkan poin terkait transaksi itu berdasarkan riwayat yang tercatat. |

**Contoh pemakaian di Controller Transaksi (Erick):**
```php
$poinModel    = new \App\Models\PoinModel();
$riwayatModel = new \App\Models\RiwayatPoinModel();

// Setelah transaksi sukses disimpan:
$poinDidapat = $poinModel->hitungPoin($totalBelanja);
if ($poinDidapat > 0) {
    $riwayatModel->prosesTambahPoin($userId, $poinDidapat, $transaksiId);
}

// Saat checkout, kalau pembeli mau pakai poin:
$berhasil = $riwayatModel->prosesPakaiPoin($userId, $poinDipakai, $transaksiId);
if (! $berhasil) {
    // saldo poin tidak cukup, batalkan proses pakai poin / tampilkan pesan error
}

// Saat transaksi di-cancel:
$riwayatModel->rollbackPoinByTransaksi($transaksiId);
```

⚠️ **Catatan penting untuk Erick:** method-method di atas **tidak mencegah pemanggilan ganda** (tidak idempotent). Kalau `prosesTambahPoin()` dipanggil 2x untuk transaksi yang sama, poin akan bertambah 2x. Modul Transaksi wajib memastikan proses ini hanya dipanggil **sekali** per transaksi (misal dengan pengecekan status sebelum/sesudah, atau flag "poin sudah diproses" di tabel transaksi).

## Bug yang Ditemukan & Workaround

**Bug:** `Model::insert()` bawaan CodeIgniter di environment project ini melempar error:
```
TypeError: CodeIgniter\Database\BaseBuilder::setBind(): Argument #1 ($key) must be of type string, int given
```
Error ini muncul konsisten setiap kali `RiwayatPoinModel` memanggil `$this->insert($data)` bawaan Model, walau `$data` sudah berupa array asosiatif dengan key string yang benar (sudah dicoba tanpa nilai `null` sama sekali, error tetap sama). Kemungkinan ada masalah di instalasi/versi framework CodeIgniter di beberapa environment tim.

**Workaround yang dipakai:** semua method insert di `RiwayatPoinModel.php` (`tambahPoin()`, `kurangiPoin()`, `rollbackPoinByTransaksi()`) memakai query builder langsung (`\Config\Database::connect()->table('riwayat_poin')->insert($data)`), **bukan** `$this->insert()` bawaan Model. Ini terbukti berhasil dan sudah dites manual (lihat hasil testing di bawah).

**Perlu dilaporkan ke tim:** bug ini kemungkinan mempengaruhi model lain juga yang memakai `$model->insert()` biasa — belum tentu ketahuan kalau belum dites di skenario yang memicu.

## Hasil Testing Manual

Dites manual (sebelum modul Transaksi Erick tersedia) lewat route sementara `test-poin` (sudah dihapus dari `Routes.php` setelah testing selesai):

- `prosesTambahPoin($userId=2, 25, null, 'Testing manual')` dipanggil 2x → saldo poin user id 2 bertambah jadi `50` (25 + 25), sesuai ekspektasi (setiap panggilan menambah 25 poin, tidak ada proteksi duplikasi — lihat catatan di atas).
- Saldo poin yang ter-update di tabel `user` **terkonfirmasi tampil dengan benar** di halaman `/mahasiswa/profile`.

## Yang BELUM Selesai / Perlu Dikerjakan Lanjut

- **Integrasi nyata ke Transaksi**: method sudah siap (lihat tabel di atas), tapi belum terpasang karena modul Transaksi (Erick) belum ada logika checkout/pembayaran — `Pesan.php` saat ini baru menampilkan daftar menu & promo bundling, belum ada keranjang/checkout.
- **Pembatasan role di halaman `/manage-poin`**: sudah otomatis terlindungi lewat `AuthFilter` (cek `login_type = backoffice`), tidak perlu kerjaan tambahan.
- **UI untuk pembeli memilih pakai poin saat checkout**: menunggu UI checkout dari modul Transaksi tersedia.

## Catatan untuk Tim

- Halaman back office fitur ini memakai pola `renderPage()` (Header + Menu + konten + Footer), **bukan** `extend()`/`section()` bawaan CodeIgniter — konsisten dengan halaman Menu yang sudah ada.
- `saldo_poin` di tabel `user` **tidak** termasuk `allowedFields` di `UserModel.php` (punya Jasmen) — sengaja tidak diubah, karena semua update `saldo_poin` dilakukan lewat query builder langsung di `RiwayatPoinModel.php`, bukan lewat `UserModel`.
